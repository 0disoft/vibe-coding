# โปรแกรม Bug Bounty

> อัปเดตล่าสุด: {{LAST_UPDATED}}

## บทนำ

ที่ {{SITE_NAME}} เราร่วมมือกับนักวิจัยด้านความปลอดภัยเพื่อสร้างสภาพแวดล้อมทางอินเทอร์เน็ตที่ปลอดภัยยิ่งขึ้น หากคุณพบช่องโหว่ด้านความปลอดภัยในบริการของเรา โปรดติดต่อเราทันที

โปรแกรมนี้ดำเนินการในฐานะ **ช่องทางการเปิดเผยช่องโหว่ (Vulnerability Disclosure Channel)** ที่เน้นการรายงานอย่างรับผิดชอบและความร่วมมือมากกว่ารางวัลที่เป็นตัวเงิน เราให้ความสำคัญและร่วมมืออย่างโปร่งใสกับนักวิจัยที่เปิดเผยช่องโหว่อย่างมีจริยธรรม

## ขอบเขต (In-Scope)

นักวิจัยสามารถทดสอบได้เฉพาะโดเมนและบริการต่อไปนี้เท่านั้น:

- เว็บไซต์อย่างเป็นทางการของ {{SITE_NAME}} และโดเมนย่อย
- แอปพลิเคชันมือถืออย่างเป็นทางการของ {{SITE_NAME}} (เมื่อมีให้บริการ)
- ปลายทาง API (API endpoints) ที่ดำเนินการโดย {{SITE_NAME}} โดยตรง

โดเมนและบริการของบุคคลที่สาม (เกตเวย์การชำระเงิน, เครื่องมือวิเคราะห์, ผู้ให้บริการโฮสติ้ง ฯลฯ) ที่ไม่ได้ระบุไว้ข้างต้น ไม่อยู่ภายใต้โปรแกรมนี้ หากคุณไม่แน่ใจว่าเป้าหมายอยู่ในขอบเขตหรือไม่ โปรดติดต่อเราก่อนทำการทดสอบ

## นโยบายรางวัล

ปัจจุบัน โปรแกรม Bug Bounty ของ {{SITE_NAME}} ยังไม่มีการมอบรางวัลที่เป็นตัวเงิน อย่างไรก็ตาม เพื่อแสดงความขอบคุณสำหรับการมีส่วนร่วมที่สำคัญ เราดำเนินการดังต่อไปนี้:

- **Hall of Fame**: แสดงรายชื่อนักวิจัยที่รายงานช่องโหว่ที่ถูกต้อง (เมื่อเปิดดำเนินการ)
- **การยอมรับต่อสาธารณะ**: ให้การยอมรับต่อสาธารณะหรือจดหมายแนะนำตัวโดยได้รับความยินยอมจากนักวิจัย
- **ลำดับความสำคัญในอนาคต**: ให้โอกาสในการเข้าร่วมก่อนหากเราเปลี่ยนเป็นโปรแกรมแบบชำระเงินในอนาคต

เราอาจนำระบบรางวัลที่เป็นตัวเงิน (Bounty) มาใช้ในอนาคตขึ้นอยู่กับงบประมาณและขนาดของบริการ และเราจะประกาศในหน้านี้หากมีการดำเนินการ

## เกณฑ์ความรุนแรง

| ความรุนแรง | CVSS | ตัวอย่าง |
|---|---|---|
| Critical | 9.0-10.0 | การเรียกใช้โค้ดจากระยะไกล (RCE), ฐานข้อมูลรั่วไหลทั้งหมด, การยึดบัญชีจำนวนมาก |
| High | 7.0-8.9 | SQL Injection, Stored XSS, การข้ามการตรวจสอบสิทธิ์ |
| Medium | 4.0-6.9 | Reflected XSS, CSRF ในการกระทำที่ละเอียดอ่อน, การเปิดเผยข้อมูล |
| Low | 0.1-3.9 | ขาดส่วนหัวความปลอดภัย (Security headers), การเปิดเผยเวอร์ชัน |

ความรุนแรงอาจปรับเปลี่ยนได้ตามผลกระทบจริง

## การรายงานช่องโหว่

### ช่องทางการรายงาน

- **อีเมล**: [{{EMAIL}}](mailto:{{EMAIL}})
- **หัวเรื่อง**: `[Security Report] Vulnerability Summary` (สรุปช่องโหว่)
- **ภาษา**: โปรดเขียนเป็นภาษาเกาหลี ภาษาอังกฤษ หรือภาษาไทย

### รูปแบบรายงาน

เพื่อช่วยให้เราวิเคราะห์และทำซ้ำปัญหาได้ โปรดรวมสิ่งต่อไปนี้:

1. ประเภทของช่องโหว่และคำอธิบายโดยละเอียด
2. ขั้นตอนเฉพาะในการทำซ้ำปัญหา (รวมถึงสคริปต์หรือบรรทัดคำสั่ง)
3. URL, ปลายทาง API หรือส่วนประกอบที่ได้รับผลกระทบ
4. โค้ดพิสูจน์แนวคิด (PoC) หรือภาพหน้าจอ

### มาตรฐานคุณภาพรายงาน

- รายงานที่ไม่สามารถทำซ้ำได้หรือขาดรายละเอียดที่เพียงพออาจไม่ได้รับการยอมรับ
- ไม่ยอมรับรายงานจากผลลัพธ์ของเครื่องสแกนอัตโนมัติ
- **รายการซ้ำ**: ช่องโหว่ที่ซ้ำกันจะให้เครดิตแก่ผู้รายงานคนแรกเท่านั้น (อ้างอิงจากเวลาประทับที่เซิร์ฟเวอร์อีเมลได้รับ)

### กระบวนการ

1. **การยืนยันการรับ**: เราจะส่งอีเมลยืนยันภายใน 72 ชั่วโมงหลังจากได้รับรายงานของคุณ
2. **การวิเคราะห์และการวางแผน**: เมื่อตรวจสอบช่องโหว่แล้ว เราจะประเมินความรุนแรงและแจ้งให้คุณทราบถึงระยะเวลาในการแก้ไขโดยประมาณ หากเราไม่สามารถดำเนินการได้ตามกำหนดเวลา เราจะอธิบายเหตุผลและแจ้งกำหนดการที่อัปเดตให้ทราบ
3. **การแก้ไขและการแจ้งเตือน**: เราจะแจ้งให้คุณทราบเมื่อแพตช์เสร็จสมบูรณ์ การแก้ไขอาจใช้เวลาหากจำเป็นต้องมีการเปลี่ยนแปลงโครงสร้างเพื่อความเสถียรของบริการ
4. **การเปิดเผยและการยอมรับ**: เมื่อแก้ไขแล้ว เราจะตัดสินใจเกี่ยวกับการเปิดเผยโดยปรึกษากับนักวิจัย รายงานที่ถูกต้องจะได้รับการยอมรับตาม 'นโยบายรางวัล' ข้างต้น
5. **การออก CVE**: สำหรับช่องโหว่ที่สำคัญ เราอาจขอออกหมายเลข CVE โดยได้รับความยินยอมจากผู้รายงาน

### นโยบายการเปิดเผยต่อสาธารณะ (Public Disclosure)

- เราแนะนำให้เปิดเผยหลังจากแพตช์เสร็จสมบูรณ์และขอให้คุณแบ่งปันรายละเอียดการเปิดเผยกับเราล่วงหน้า
- หากไม่มีการดำเนินการที่เหมาะสมภายใน **60 วัน** หลังจากรายงาน ผู้รายงานมีสิทธิ์เปิดเผยรายละเอียดช่องโหว่ในลักษณะที่ตกลงกันร่วมกัน (อย่างไรก็ตาม เราอาจขอปรับเปลี่ยนกำหนดการสำหรับปัญหาที่ซับซ้อน)
- ในกรณีเร่งด่วนที่สังเกตเห็นการโจมตีจริง เราอาจประสานงานกับคุณเพื่อเปิดเผยก่อนกำหนด

## นโยบายและแนวทางการทดสอบ

โปรดปฏิบัติตามแนวทางต่อไปนี้เพื่อการทดสอบช่องโหว่ที่ปลอดภัย

### กิจกรรมที่อนุญาต

- ทดสอบช่องโหว่โดยใช้บัญชีที่คุณเป็นเจ้าของหรือบัญชีทดสอบที่คุณสร้างขึ้น
- **การตรวจสอบขั้นต่ำ**: เข้าถึงเฉพาะข้อมูลขั้นต่ำที่จำเป็นเพื่อพิสูจน์ช่องโหว่ หากคุณเข้าถึงข้อมูลที่ละเอียดอ่อนของผู้อื่นโดยไม่ได้ตั้งใจ ให้หยุดทันทีและรวมเฉพาะข้อมูลที่ปิดบังไว้ในรายงานของคุณ

### สภาพแวดล้อมการทดสอบ

- **ขอระบุบัญชีทดสอบ**: หากคุณต้องการบัญชีทดสอบ คุณสามารถขอได้ที่ [{{EMAIL}}](mailto:{{EMAIL}})
- **การสแกนอัตโนมัติ**: อนุญาตให้สแกนแบบเบาๆ ได้ แต่การสแกนที่มีโหลดสูงซึ่งสร้างคำขอมากเกินไปต่อวินาทีหรือส่งผลกระทบต่อคุณภาพบริการ ต้องมีการประสานงานล่วงหน้า

### กิจกรรมต้องห้าม (นอกขอบเขต)

กิจกรรมต่อไปนี้เป็นสิ่งต้องห้ามอย่างเคร่งครัดและอาจไม่ได้รับการคุ้มครองตามกฎหมายหากมีการละเมิด:

- การรัน **การสแกนอัตโนมัติที่มากเกินไป** (ในระดับที่ทำให้เกิดภาระต่อบริการ) โดยไม่มีการประสานงานล่วงหน้า
- กิจกรรมใดๆ ที่จงใจทำให้ทรัพยากรเซิร์ฟเวอร์หมดลง (CPU, หน่วยความจำ, ดิสก์, แบนด์วิดท์เครือข่าย)
- การเข้าถึงหรือแก้ไขบัญชี ข้อมูล หรือข้อมูลส่วนบุคคลของผู้ใช้รายอื่น
- วิศวกรรมสังคม (Social engineering) (ฟิชชิ่ง ฯลฯ) หรือการโจมตีความปลอดภัยทางกายภาพ

### นอกขอบเขตอย่างชัดเจน (Out-of-Scope)

- ช่องโหว่ที่พบในบริการหรือโครงสร้างพื้นฐานของบุคคลที่สาม
- ความปลอดภัยทางกายภาพ, ระบบ HR, เครือข่ายภายใน
- ช่องโหว่ที่เปิดเผยต่อสาธารณะแล้วหรือรายงานที่ซ้ำกัน
- ปัญหาที่เกิดจากการส่งสแปมหรืออีเมลฟิชชิ่งเท่านั้น

### ช่องโหว่ที่มีความเสี่ยงต่ำ (ไม่ยอมรับ)

รายการต่อไปนี้ได้รับการยกเว้นจากโปรแกรมเนื่องจากมีความเสี่ยงด้านความปลอดภัยต่ำหรือเป็นพฤติกรรมที่ตั้งใจไว้:

- CSRF ที่มีผลกระทบต่ำ เช่น CSRF ออกจากระบบ
- Clickjacking บนหน้าเพจที่ไม่มีข้อมูลที่ละเอียดอ่อน
- การเปิดเผยเวอร์ชันง่ายๆ (banner grabbing)
- การตั้งค่าความปลอดภัยที่ขาดหายไปโดยไม่มีเส้นทางการโจมตีที่พิสูจน์แล้ว (เช่น ขาดส่วนหัวความปลอดภัยโดยไม่มีผลกระทบโดยตรง นโยบายการส่งอีเมลที่ไม่ได้กำหนดค่า ฯลฯ)
- เปิดใช้งานการปอกข้อมูลอัตโนมัติของเบราว์เซอร์

อย่างไรก็ตาม แม้แต่รายการข้างต้นก็อาจได้รับการประเมินหากมีการเชื่อมโยงกับช่องโหว่อื่นๆ เพื่อพิสูจน์สถานการณ์การโจมตีจริง

### การคุ้มครองนักวิจัย (Safe Harbor)

หากคุณวิจัยและรายงานช่องโหว่โดยสุจริตและปฏิบัติตามนโยบายนี้ เราสัญญาสิ่งต่อไปนี้ **เท่าที่กฎหมายที่บังคับใช้จะอนุญาต**:

1. เราถือว่ากิจกรรมการวิจัยของคุณเป็นการวิจัยด้านความปลอดภัยที่ได้รับอนุญาตและจะไม่ดำเนินการทางกฎหมายแพ่งหรืออาญาใดๆ กับคุณ
2. เราจะไม่รายงานคุณต่อหน่วยงานบังคับใช้กฎหมายหรือยื่นคำร้องเรียนโดยสมัครใจ
3. หากบุคคลที่สามเริ่มดำเนินการทางกฎหมายเกี่ยวกับกิจกรรมการวิจัยของคุณ เราจะให้การสนับสนุนภายในขอบเขตที่เหมาะสม เช่น การจัดเตรียมเอกสารที่พิสูจน์ว่าคุณเป็นนักวิจัยที่ปฏิบัติตามกฎระเบียบ

อย่างไรก็ตาม Safe Harbor ไม่สามารถใช้ได้ในกรณีต่อไปนี้:

- การละเมิดกิจกรรมต้องห้ามในเอกสารนี้อย่างชัดเจน
- การทดสอบระบบหรือโครงสร้างพื้นฐานของบุคคลที่สามที่ไม่ได้รับอนุญาตซึ่งอยู่นอกเหนือการควบคุมของเรา
