C 코드와 연동하는 프로젝트에서 이전에는 조용하던 누수가 에러로 보이기 시작할 수 있다. 환경 변수 ASAN_OPTIONS=detect_leaks=0 으로 비활성화 가능. - go.mod에 ignore 지시어가 추가됐다. 이걸로 벤치마크용, 실험용, 생성 코드 등 패턴에 섞이고 싶지 않은 디렉터리를 명시적으로 무시할 수 있다. - go doc -http 옵션이 생겨서, 로컬에서 브라우저 기반 문서 서버를 띄울 수 있다. - go version -m -json 으로 빌드 정보(runtime/debug.BuildInfo)를 JSON으로 뽑아낼 수 있어, 배포된 바이너리 추적이 편해진다. - go-import 메타 태그에 서브디렉터리를 직접 명시하는 형식을 지원해서, 하나의 저장소를 여러 모듈로 나눌 때 더 유연해짐. 3. 런타임 - GOMAXPROCS 기본값이 컨테이너 환경을 인식하게 바뀌었다. 리눅스에서 cgroup CPU 제한이 있으면, 논리 CPU 개수가 아니라 그 제한에 맞춰 기본값을 잡는다. 또, CPU 개수나 cgroup 제한이 바뀌면 런타임이 GOMAXPROCS를 주기적으로 갱신한다. GOMAXPROCS 환경 변수나 runtime.GOMAXPROCS 호출로 값을 고정하면 이 동작은 꺼지고, GODEBUG=containermaxprocs=0, updatemaxprocs=0 으로도 끌 수 있다. - 새 실험적 GC(greenteagc)가 들어왔다. GOEXPERIMENT=greenteagc 로 켜며, 작은 객체를 많이 쓰는 서비스의 GC 오버헤드를 크게 줄일 수 있도록 설계되어 있다. - runtime/trace에 FlightRecorder API가 생겨서, 항상 거대한 트레이스를 남기지 않고도, 문제가 생겼을 때 최근 몇 초만 덤프하는 식의 사용이 가능해졌다. - 처리되지 않은 패닉 메시지 포맷이 약간 바뀌어, recover 했다가 다시 panic 한 경우 메시지가 더 간결하고 의미가 분명해졌다. 4. 컴파일러 - 오래된 버그 하나를 확실히 잡았다. nil 포인터 체크를 너무 늦게 해서, 엄밀히는 잘못된 코드가 운 좋게 잘 돌아가던 케이스가 있었는데, 이제는 제대로 panic이 난다. 예를 들어 os.Open 에러를 검사하기 전에 반환값을 먼저 사용하던 코드는 1.25에서 갑자기 panic 할 수 있다. 해결책은 단순히 에러를 바로 검사하도록 코드 순서를 고치는 것. - 디버그 정보가 DWARF 버전 5로 올라가서, 큰 바이너리에서 링크 시간과 디버그 정보 크기가 줄어든다. - 슬라이스 백킹 배열을 스택에 더 공격적으로 올려서 성능을 개선하지만, unsafe.Pointer 오용은 더 빨리 터질 수 있다. 5. 표준 라이브러리 주요 포인트 - testing/synctest: 동시성 테스트용 새 패키지. Test 함수는 가상의 시간과 고루틴 버블 안에서 테스트를 돌리고, Wait 함수로 모든 고루틴이 블록될 때까지 기다릴 수 있다. 1.24에서는 GOEXPERIMENT=synctest 실험으로 들어왔고, 1.25에서 정식 패키지가 됐다. 단, 예전 실험 API는 향후 1.26에서 제거 예정이라 새 코드는 정식 API에 맞춰야 한다. - encoding/json/v2: GOEXPERIMENT=jsonv2 를 켜면, 새로운 JSON 구현이 활성화된다. encoding/json/v2 패키지와 encoding/json/jsontext 패키지가 추가되고, 기존 encoding/json 패키지 내부 구현도 새 엔진으로 바뀐다. 마샬·언마샬 결과는 유지하려고 했지만, 에러 메시지 텍스트나 세부 동작이 달라질 수 있다. 대신 디코딩 성능은 크게 향상. - crypto 계열: MessageSigner, SignMessage 같은 새 인터페이스와 헬퍼가 생겨서, 해시를 직접 처리하고 싶은 서명 구현에 더 잘 맞는다. crypto/ecdsa, crypto/ed25519, crypto/rsa 등에서 키 인코딩과 성능이 다듬어졌고, crypto/elliptic 의 숨겨진 메서드는 정리됐다. - crypto/tls: TLS 1.2에서 SHA-1 서명 알고리즘을 기본적으로 거부한다. RFC 9155를 따른 것이라, 진짜로 예전식 서버와 통신해야 하는 특수 환경이 아니라면 바람직한 변화다. 필요하면 GODEBUG=tlssha1=1 로 임시 우회는 가능하다. - go/ast, go/parser: 오래된 Object, Package 기반의 타입 정보를 전제로 하던 API들(FilterPackage, PackageExports, MergePackageFiles, ParseDir 등)이 deprecated 상태가 됐다. 대신 go/packages, go/types 기반의 현대적인 패턴을 쓰라는 신호다. - hash, hash/maphash, io/fs, log/slog, net, os, reflect, regexp, net/http 등 곳곳에 작은 기능 추가가 있다. 특히 net/http에는 CrossOriginProtection이라는 CSRF 방어용 헬퍼가 들어와서, 토큰 없이 Fetch 메타데이터로 요청을 차단하는 방식을 쓸 수 있다. | 실무적인 정리 - 컨테이너에서 도는 서비스는, 1.25부터 GOMAXPROCS 기본값이 CPU 제한을 따라간다. Kubernetes에서 CPU limit를 타이트하게 걸어둔 서비스는, 갑자기 고루틴 스케줄링이 달라진 것처럼 느껴질 수 있으니 모니터링 지표를 한번 확인하는 게 좋다. 필요하면 GODEBUG로 이 기능을 꺼두고, 직접 GOMAXPROCS를 설정하는 전략도 가능. - C와 연동하며 asan 빌드를 쓰는 프로젝트는, 이제 C 쪽 메모리 누수도 에러로 보고된다. 새 에러를 억지로 숨기지 말고, 가능하면 이 기회에 C 코드 누수도 같이 청소하는 게 장기적으로 이득. - 오래된 nil 체크 버그에 의존하던 코드가 있다면 지금이 고칠 마지막 기회라고 보면 된다. 에러 반환 후 바로 에러를 검사하는 패턴으로 통일해두면, 이 버전뿐 아니라 이후에도 안전하다. - 동시성 테스트를 진지하게 하고 싶다면 testing/synctest 도입을 고려해볼 만하다. 고루틴 폭발이나 타이밍 의존 버그를 재현·고정하기가 훨씬 수월해진다. - JSON 처리에 병목이 있는 서비스는 GOEXPERIMENT=jsonv2 를 켜고 스테이징 환경에서 한번 돌려보는 걸 추천한다. 호환성 문제가 없는지 확인한 뒤에 프로덕션에 올리면, 별도 라이브러리 없이도 꽤 큰 성능 이득을 볼 수 있다. - go/ast·go/parser의 deprecated API를 쓰고 있다면, 장기적으로 go/packages 기반 구조로 리팩터링할 계획을 잡는 게 좋다. 고언어 자체가 오래된 타입 시스템 접근 방식을 청산하는 방향으로 가고 있다. |

요약하면

- 언어 차원에서 코딩 습관이 바뀌어야 할 정도의 변화는 1.25의 nil 체크 버그 수정 정도
- 런타임·운영 관점에서 크게 바뀐 건 1.25의 컨테이너 인식 GOMAXPROCS
- 기능 측면에서 눈여겨볼 건 1.24의 crypto 정책 강화, 1.25의 testing/synctest·encoding/json v2·새 go 모듈 도구들

그리고 기간 바깥이긴 하지만, 1.23.0에서 도입된 변화들도 지금 1.22에서 1.25로 점프할 거라면 반드시 읽어볼 가치가 있다. 타임라인 기준으로는 제외했지만, 실제 마이그레이션 계획 세울 땐 1.22 → 1.23 → 1.24 → 1.25 순서로 릴리스 노트를 한번씩 훑어보는 걸 추천할게.
