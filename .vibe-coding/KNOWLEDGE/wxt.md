# wxt 주요 변경점 요약 (2024-11-13 이후)

먼저 결론부터 말하면, 내가 지금 확인할 수 있는 범위에서
2024-11-13 이후 wxt 코어 패키지의 릴리스는 `0.19.22`부터 `0.20.11`까지야.
그 전에 `0.19.21` 같은 패치가 있을 수는 있는데, GitHub 릴리스 페이지를 더 내려보는 데 한계가 있어서 11월 후반부 일부 패치는 빠졌을 가능성이 있음.
대신, 내가 실제로 읽을 수 있었던 릴리스 노트 기준으로만 확실한 내용만 정리할게. 🙇‍♂️

## 2. 코드 짤 때 알아둬야 하는 변화만 골라서 정리

질문에서 말한 것처럼

- 단순 버그 수정
- 성능 개선
- 문서/CI/의존성 버전업 같은 건 과감히 버리고
- “내 코드를 바꿔야 할 수도 있는 변화” 위주로만 뽑았어.

정리해보면

- 0.19.x 라인에서는 “대규모 API 변경·deprecate”보다는
  타입/빌드/런타임 버그 수정과 툴 호환성(Vite, publish-browser-extension)이 대부분이었고
- 0.20.x 라인은 실제로 큰 변경점은 `0.20.0`에 몰려 있을 가능성이 높은데,
  내가 지금 읽을 수 있는 릴리스 노트 일부만으로는 정확한 deprecate/마이그레이션 내용을 재구성하기가 어렵다 보니
  “0.20.0의 구조적 변경”은 공식 업그레이드 가이드를 그대로 따라가는 걸 강하게 추천할 수밖에 없어.([wxt.dev][5])

솔직히 말하면, “반드시 알아야 하는 변경” 관점에서 이 구간의 패치 릴리스 대부분은

- 내가 이미 쓰던 코드를 깨뜨리기보다는
- 잘못되던 동작을 바로잡거나
- 새 Vite 버전·툴 버전과 잘 어울리게 만들어주는 쪽에 가깝고

실제로 마이그레이션 난도가 확 튀는 지점은

- `0.19.0` 출시 시점의 구조 변화
- `0.20.0` 출시 시점의 구조/설정 변화

이 두 군데야. 이 둘은 wxt 공식 문서의 “Upgrading” 섹션을 그대로 따라가면서 점프하는 게, 개별 릴리스 노트를 다 쪼개 읽는 것보다 훨씬 안전해 보여.([wxt.dev][5])

## 3. 현실적인 업그레이드 전략 제안

wxt로 브라우저 확장 하나만 쓰는 게 아니라, 앞으로 여러 프로젝트에서 계속 쓸 생각이면 이렇게 정리하는 게 좋을 것 같아.

- 지금 프로젝트가 `0.19.x` 이하라면
  - 바로 `0.20.11`로 점프하는 대신
  - 문서에 있는 “Upgrading 0.19 → 0.20” 가이드를 한 번에 따라가고
  - 그 뒤에 0.20.1 ~ 0.20.11은 “패치 레벨”로 다 올리는 게 마이그레이션 스트레스가 덜함.

- Vite 버전 전략
  - Vite 6.0.9+를 쓰고 있다면, `0.19.25`의 제한을 의식해야 해서
  - 아예 “wxt 0.20.x + Vite 7 조합”을 목표로 올리는 게 깔끔할 가능성도 있음.([GitHub][3])

- 테스트 환경
  - `jsdom` / `happy-dom` 기반 테스트를 무겁게 쓰는 프로젝트라면
  - `0.20.11` 기준의 `wxt/testing` 모듈 구조를 한 번 정리해두고
  - 앞으로도 그 패턴을 템플릿처럼 복붙해서 쓰는 쪽이 유지보수에 유리할 거야.([GitHub][4])
