## 2. 진짜 코드에 영향을 주는 변화만 뽑아서 정리

### 2-1. 새 기능 / 의미 있는 동작 변화

"실제 구현에 써먹거나, 코드 구조를 바꾸게 만들 만한 것"만 골랐어.

| 3.14.0                      | 애너테이션 의미론    | PEP 649, PEP 749. 기본적으로 애너테이션이 "지연 평가되는 함수"로 저장됨. 더 이상 무조건 문자열로만 저장되지 않고, `annotationlib.get_annotations()` 같은 헬퍼로 값을 원하는 형식으로 평가할 수 있음. `from __future__ import annotations`를 쓴 모듈은 예외적으로 기존 스타일 유지 | 프레임워크나 DI 컨테이너처럼 `__annotations__`를 직접 파는 라이브러리는 수정이 필요할 수 있다. 이제 단순 딕셔너리에서 문자열·타입 객체가 섞여 있는 게 아니라, 지연 평가용 오브젝트가 들어오기 때문이다. 대신 "앞에서 정의하지 않은 타입을 문자열로 감싸서 쓰는 꼼수"는 더 이상 필수가 아님. 대부분의 forward reference가 자연스럽게 동작한다. 타입 정보를 읽고 싶으면 `annotationlib`를 사용하는 쪽으로 코드를 옮기는 게 안전하다.           |

| 3.14.0                      | 문자열 / 템플릿      | PEP 750. `t"..."` 템플릿 문자열 도입. 안전한 템플릿 작업을 위해 새 `string.Template` 계열과 연동                                                                                                                                                                                  | Jinja 같은 템플릿 엔진을 쓸 만큼은 아닌데, `str.format` / f-string이 조금 무겁거나 위험하게 느껴질 때 `t""`가 대안이 될 수 있다. 특히 "사용자 입력을 섞은 메시지"를 만들 때 포맷 문자열 공격을 피하기에 좋다.                                                                                                                                                                                                                                |
| 3.14.0                      | 디버깅 / 리모트 실행 | PEP 768. 외부 디버거와 안전하게 상호작용하는 인터페이스, `sys.remote_exec()` 제공                                                                                                                                                                                                 | 원격 Python 프로세스에 디버깅 스니펫을 실행시키거나, 런닝 프로세스 안에서 코드를 주입해야 하는 상황(예: 원격 디버깅, 운영 중인 서비스에 안전한 인스펙션)을 다룰 때 쓰인다. 자체 디버거/관리 콘솔을 만들 생각이 있다면 이 인터페이스를 채택하는 게 좋다.                                                                                                                                                                                      |
| 3.14.x                      | free-threaded 상태   | PEP 779 등으로 free-threaded Python이 생태계에서 점점 "정식 옵션"에 가까워짐                                                                                                                                                                                                      | 휠 태그(cp313t 등), 빌드 설정, CI 매트릭스를 설계할 때 "no-GIL 빌드도 지원할지"를 결정해야 한다. 고성능 라이브러리를 만드는 쪽이라면 지금부터 free-threaded 환경에서 테스트를 돌리기 시작하는 게 안전하다.                                                                                                                                                                                                                                   |

이 표에 없는 3.12.x, 3.11.x, 3.10.x, 3.9.x 마이너 버전들은 공식 릴리스 노트 기준으로 "버그·보안·빌드·문서 수정" 위주이고, 새 언어 기능·표준 라이브러리 API 추가는 사실상 없다고 보면 된다.

즉, 실무 코드에 영향을 줄 정도의 의미 있는 변화는 거의 전부 3.13·3.14 쪽에 몰려 있다.

### 2-2. 삭제·Deprecated 중심 정리 (진짜 깨질 수 있는 부분)

아래는 "그냥 놔두면 경고가 뜨거나, 아예 ImportError/AttributeError가 나는" 것들만 모은 표야.

| 버전   | 대상                                                                                                                                                                                                                          | 왜 deprecated / 제거됐나                                                                                                                                                     | 대체 / 대응 방법                                                                                                                                                                                                                                                                                                                                                              |

## 3. 이걸 보고 나면 어떤 버전을 타깃으로 잡는 게 좋냐

너 입장에서 "실제로 개발할 때 신경 쓸 버전"을 정리하면 대략 이렇게 보일 거야.

- 새 프로젝트 시작
  가능하면 3.13 계열 이상을 기본 타깃으로 잡는 게 좋아 보인다.
  이유는
  - 지원 기간이 길고
  - no-GIL / JIT / 새로운 타이핑 기능 등 "앞으로 Python이 가고 싶은 방향"이 모두 이쪽에 달려 있기 때문.

- 타입·프레임워크를 깊게 쓰는 서버·툴링
  - 애너테이션 의미론이 크게 바뀐 3.14를 언제부터 지원할지 한 번 전략을 정해야 한다.
  - 직접 `__annotations__`를 후비는 라이브러리를 만든다면, 3.13과 3.14에서 모두 잘 돌아가게 하려면 `annotationlib` 헬퍼를 기준으로 재설계하는 게 현실적이다.

- 레거시 지원
  - 3.12는 이미 "마지막 full 유지보수 릴리스"를 지났고, 보안 패치 모드로 들어갔다. 새 기능을 기대할 수는 없지만, 지금 돌아가는 코드를 당장 깨지는 일은 거의 없다.
  - 3.10·3.9는 사실상 수명 말기 보안 브랜치라, 장기적으로는 3.11 이상으로 올릴 이주 계획을 세우는 게 안전하다.

요약하면

- "최소 요구 버전"은 3.11 정도에 두고
- 새 기능을 적극 활용할 서비스·툴 체인은 3.13 이상
- 타입·애너테이션 트릭을 많이 쓰는 라이브러리는 3.14 전후로 별도 브랜치 전략

이 정도 그림이면, 앞으로 몇 년은 크게 흔들리지 않을 거야.
