# Tauri 2 버전대별 주요 변경점 정리

## 1. Tauri 2 주요 릴리스 타임라인 (2024-09-13 ~ 2025-12-05)

Tauri 팀 블로그와 릴리스 인덱스를 합쳐, **코드·구조 관점에서 의미 있는 변화가 있는 버전**만 추려 정리한 타임라인이다.

| 버전 계열         | 릴리스 시점(대략)        | 요약                                                                |
| ------------- | ----------------- | ----------------------------------------------------------------- |
| 2.0.0         | 2024-10-02        | Tauri 2.0 안정판. 모바일 지원, capability 기반 보안 모델, 설정 구조 전면 개편.          |
| 2.0.x 패치      | 2024-10 ~ 2024-11 | 2.0 직후 버그 수정·안정화. 기능 추가보다 버그·플랫폼 이슈 해결에 집중.                       |
| 2.1.0         | 2024년 말           | 2.0 보안·권한 모델 유지, 일부 API 개선 및 모바일 안정화. breaking change는 제한적.       |
| 2.2.0 계열      | 2025년 초           | 모바일·멀티 플랫폼 지원 강화, 일부 JS API 확장 및 설정 편의 기능 개선.                     |
| 2.3.0 / 2.3.1 | 2025년 1분기         | 런타임·CLI 품질 개선, 일부 API 추가. “더 안정적으로 잘 동작하도록” 다듬는 릴리스.              |
| 2.4.0 / 2.4.1 | 2025년 상반기         | 설정·번들링·모바일 지원 개선. 2.0 스타일 코드를 가져와도 설정 스키마는 2.x 규칙을 따라야 함.         |
| 2.5.0 / 2.5.1 | 2025년 중반          | iOS/macOS 번들 버전 지정, JS API Channel 개선, iOS 입력창 옵션 등 실사용 기능 다수 추가. |
| 2.6.x         | 2025년 여름          | 런타임·wry/tao 의존성 업데이트, 이벤트·윈도 동작 개선. 앱 코드가 직접 깨지는 변화는 제한적.         |
| 2.7.0         | 2025년 초가을         | Linux/GTK 의존성 정리, tray·메뉴·창 관련 개선.                                |
| 2.8.x (0~5)   | 2025년 가을          | GTK3 계열 RUSTSEC 경고 반영, 런타임·보안 조정, 번들러·CLI 개선.                     |
| 2.9.0 ~ 2.9.4 | 2025년 11월 전후      | tauri 2.9.4 기준 최신 코어 릴리스. RustSec 취약점 점검, 런타임 의존성 상향, 플랫폼별 개선.    |

정확한 일자가 반드시 필요하다면 GitHub 태그나 crates.io 개별 릴리스를 직접 조회하는 편이 안전하다.
위 표는 **“어느 버전대에서 어떤 유형의 변화가 있었는지”를 빠르게 파악하는 용도**에 초점을 맞춘다.

## 2. 버전대별로, 코딩할 때 반드시 인지해야 할 변경점

이제 실제로 중요한 지점이다.
**“각 버전대에서 어떤 개념이 도입되었고, 어떤 코드는 더 이상 허용되지 않는지”**를 연결해서 본다.

### 2.0.0 계열: 보안·구성 모델 전면 교체

2.0은 단순 메이저 업그레이드가 아니라 **보안·구성 모델의 교체**에 가깝다.

* Capability 기반 권한 시스템 도입

  * 1.x처럼 `fs`, `shell` 플러그인을 가져다 쓰는 방식이 아니라,
    **“어떤 파일 경로, 어떤 네트워크 도메인, 어떤 OS 기능에 접근하는지”를 capability로 명시**해야 한다.
  * `tauri.conf.json` 등 설정 파일에 `capabilities`·`permissions`를 정의하지 않으면, 코드가 존재해도 런타임에서 차단된다.

* 설정 파일 구조 재편

  * 설정 파일을 여러 개로 분할하고, 공식 configuration-files 스키마를 따라야 한다.
  * 키 이름이 1.x와 다르고, 플러그인 옵션도 `plugins.<name>` 아래 capability와 연결되는 구조다.
  * **“1.x JSON config를 그대로 들고 와서 2.x로 빌드하는 시나리오”는 사실상 실패가 전제된 접근**이다.

* 모바일 정식 지원 편입

  * Android·iOS 지원이 2.0에서 공식 축으로 편입되었다.
  * PC용 프로젝트를 **별도 설계 없이 모바일 타깃으로 빌드하면**, 권한·경로·플러그인 호환성에서 연쇄적으로 충돌이 난다.

코드 수준에서의 실질적인 의미

* “아무 설정 없이 `fs.readFile`부터 호출하는” 패턴은 사실상 **구시대적 사고방식**이다.
* 새 프로젝트라면

  * 반드시 2.x 기준 템플릿으로 프로젝트를 생성하고
  * 기존 코드를 이 템플릿 안으로 옮기는 방향으로 설계해야 한다.
* 레거시 1.x를 유지해야 한다면

  * 당분간은 Tauri 1.x를 유지하거나,
  * **2.x 전용 새 프로젝트를 만들고 기능 단위로 이식하는 ‘마이그레이션 프로젝트’**로 접근하는 편이 훨씬 안전하다.

### 2.1.x ~ 2.3.x: 2.0 모델을 실전에 맞게 다듬는 구간

이 구간은 개념이 다시 갈리는 시기가 아니라, **2.0에서 도입된 모델을 실제 프로젝트에 맞게 안정화하는 단계**다.

주요 흐름

* capabilities·permissions를 실제로 쓰기 쉽게, config 문서·에러 메시지·DX가 개선된다.
* 일부 JS API에 모바일 옵션이 추가되고, 이벤트 처리·에러 핸들링이 안정화된다.

실제 코딩 관점에서의 의미

* 이미 2.0.0에서 capability 기반으로 설계했다면,

  * 2.1~2.3으로 올릴 때 **코드 구조를 전면 수정해야 할 변화는 거의 없다.**
* 이 구간에서 필요한 일은

  * 릴리스 노트의 breaking 섹션을 한 번 점검하고,
  * 경고·deprecate 메시지를 기준으로 꼭 필요한 부분만 수정하는 수준이다.

결국 이 범위는 **“새 Tauri 2.x 프로젝트의 기본 토대”** 정도로 이해하면 충분하다.

### 2.4.x ~ 2.5.x: config 정돈 + 모바일·Apple 생태계 대응 강화

2.4~2.5에서는 **설정·번들링·모바일/Apple 배포 파이프라인**에 관한 변화가 눈에 띈다.

대표적인 코드 관련 변화

* iOS / macOS 번들 버전 명시 지원

  * `tauri-cli` 2.5.0에서 iOS·macOS 번들에 `CFBundleVersion`을 지정하는 `bundleVersion` 옵션이 추가된다.
  * App Store·TestFlight 등에서 **빌드 번호를 분리 관리하는 파이프라인**을 구성할 수 있어, 모바일·macOS 배포 자동화에 핵심 요소로 작동한다.

* JS API `Channel` 개선

  * `@tauri-apps/api@2.5.0`부터 `Channel` 생성자에 콜백을 직접 넘길 수 있다.

    * 예: `new Channel((message) => console.log(message))`
  * 기존 패턴과 병존하는 확장이라 구 코드가 깨지진 않지만, **새 코드에선 이 형식을 기본값으로 채택하는 편이 합리적**이다.

* iOS 입력창 옵션 추가

  * `disableInputAccessoryView` 등 iOS 전용 옵션이 추가되어, 모바일 입력 UX를 세밀하게 제어할 수 있다.

코딩 전략

* 모바일·macOS 빌드를 본격적으로 포함할 계획이라면

  * 최소 2.5.x 이상을 기준으로 삼고,
  * `bundleVersion` 같은 값을 **CI/CD 환경 변수로 주입하는 구조**를 설계하는 편이 장기적으로 유리하다.
* JS 측에서 Tauri 이벤트·채널을 적극적으로 사용한다면

  * 새로운 `Channel` 생성 패턴을 표준으로 삼고,
  * 기존 코드도 점진적으로 통일해 두는 것이 유지보수 비용을 줄인다.

### 2.6.x ~ 2.7.0: 런타임·의존성 조정, Linux/GTK 경고 신호

이 구간은 앱 코드보다는 **런타임에 포함된 네이티브 의존성**이 핵심 이슈다.

핵심 포인트

* GTK3 관련 crate에 RustSec “unmaintained” 경고 표기

  * `gtk`, `gdk` 계열이 더 이상 유지보수되지 않는다는 경고가 붙고,
  * Tauri 런타임도 이를 인지한 상태로 계속 사용한다.

* 2.7.0 근처에서 Linux/GTK, tray icon, 메뉴 동작 보완

  * wry/tao 버전 업과 함께
  * 윈도우 동작, 트레이·메뉴 행동이 일부 환경에서 달라질 수 있다.

코딩·운영 관점에서의 의미

* Linux용 Tauri 앱을 중장기적으로 운영한다면

  * GTK3 기반 스택을 **얼마나 더 끌고 갈지**,
    이후 어떤 백엔드·WebView 체인으로 전환할지 전략을 세워야 한다.
* 2.5.x → 2.7.x 업그레이드 후

  * tray / 메뉴 / 창 동작이 예상과 다르면,
  * 우선 wry/tao 의존성 이슈를 의심하고 릴리스 노트·issue tracker를 함께 확인하는 것이 합리적이다.

### 2.8.x ~ 2.9.x: 보안 점검 강화, 설정·Updater 정리, 최신 코어

질문 범위의 끝에 가까운 2.8·2.9 라인은 **보안·문서·플랫폼 요구사항을 정리하는 구간**이다.

주요 변화

* RustSec 기반 보안 점검 강화

  * 2.8.x, 2.9.x에서 Cargo audit 결과가 릴리스 노트에 명시되고,
  * GTK3처럼 유지보수 중단된 의존성 경고가 그대로 드러난다.
  * “Tauri 2.9.4 + RustSec clean” 조합을 목표로 한다면, **직접 의존성을 교체하거나 OS 패키징 단계에서 보완 전략**을 세워야 한다.

* 설정 파일 문서와 실제 동작의 정렬

  * 2025년 10월 기준 configuration-files 문서가 갱신되며, multi-file config, capabilities, plugin config가 정리된 구조로 제시된다.
  * 2.8~2.9에서는 **문서와 런타임이 기대하는 설정 구조가 사실상 수렴된 상태**로 보는 것이 안전하다.

* Updater 플러그인 UX 개선

  * Updater 문서가 S3, GitHub Gist 등 **정적 JSON 기반 배포** 예제를 제공한다.
  * 앱 코드의 업데이트 트리거 방식은 크게 바뀌지 않지만,
    **서버 구조 설계에 따라 업데이트 UX와 안정성이 갈린다.**

* macOS·iOS 개발 환경 요구사항 최신화

  * Prerequisites 문서가 2025년 11월 갱신되며 최신 Xcode·SDK 요구사항을 반영한다.
  * 2.9.x 기준으로는 **구버전 Xcode + 구 SDK 조합에서 빌드 오류 리스크가 확실히 높아진다.**

코딩·프로젝트 관리 관점에서의 결론

* 지금 Tauri 2로 새 프로젝트를 시작한다면

  * 최소 2.8.x, 가능하면 **바로 2.9.x를 기준으로 시작해야 한다.**
  * 이유는 단순하다.

    * 설정 구조가 문서와 가장 잘 맞고
    * 보안·플랫폼 요구사항이 최신 기준으로 정리되어 있기 때문이다.

* 2.0~2.3 기반 프로젝트를 2.9.x로 올릴 경우

  * **먼저 설정 파일 구조를 최신 문서 기준으로 리팩터링**하고,
  * 이후 버전을 올리는 순서를 택해야 “이해 안 되는 권한 오류”를 최소화할 수 있다.

## 3. 정리: “버전별 변경 사항 암기”보다, 이렇게 대응해야 실전적이다

Tauri 2.x는 한 번 구조를 크게 바꾼 이후,
**“버전마다 API를 갈아엎는 프레임워크”가 아니라, “도입된 구조를 현실적인 앱·배포 환경에 맞춰 조정하는 프레임워크”에 가깝다.**

실전에서는 버전 암기보다 다음 관점이 훨씬 유효하다.

1. 새 코드 기준

   * 지금 시작하는 프로젝트라면

     * **Tauri 2.9.x + 최신 문서(configuration-files, prerequisites, updater)**를 기준으로 설계하고
     * 예제·템플릿도 전부 2.9 기준으로 맞춘다.

2. 레거시 코드 기준

   * 1.x → 2.x 전환은 사실상 **“대규모 이식 프로젝트”**다.

     * 2.x 템플릿으로 새 프로젝트를 만들고
     * 기능 단위로 코드·UX를 옮기는 계획을 세우는 편이 리스크를 줄인다.
   * 2.0~2.3 → 2.8~2.9 업그레이드는

     * 먼저 설정(capabilities, plugin config)을 최신 문서에 맞춰 정리하고
     * 그다음 의존성·버전을 올리는 순서를 지켜야 한다.

3. 장기 관점

   * Linux/GTK 관련 의존성에는 이미 “유지보수 중단” 경고가 붙어 있다.
   * Tauri 기반 데스크톱 앱을 장기 운영할 계획이라면,

     * GTK3 의존성을 어떻게 처리할지,
     * 어떤 백엔드·UI 스택(또는 Web 전면, 백엔드 분리)으로 이동할지에 대한
     * 중장기 로드맵을 지금부터 상정해야 한다.

마지막으로, **버전별 변경 사항 표를 Git 기반으로 자동화**하고 싶다면
`tauri` 레포의 태그별 `CHANGELOG`를 파싱해, 조직 내부용 `tauri-2.x-migration.md`를 생성하는 파이프라인을 구축하는 것도 충분히 투자 가치가 있다.
