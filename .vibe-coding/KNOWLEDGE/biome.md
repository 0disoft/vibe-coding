#### 1.9.3 (2024-10-01)

이 버전은 린트 규칙이 꽤 많이 늘어나서, 팀 규칙에 따라 코드가 빨간색으로 물들 수도 있는 구간입니다.

| 범주               | 변경 내용                                                                       | 영향                                                                 |
| ------------------ | ------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| Linter - CSS       | `noDescendingSpecificity` 추가. 더 구체적인 선택자가 덜 구체적인 선택자 아래에 오는 경우 경고. | Tailwind 없이 직접 CSS를 쓰는 프로젝트라면 선택자 순서 때문에 경고 증가 가능. |
| Linter - JS        | `noNestedTernary` 추가. 삼항연산자 중첩을 막음.                                  | 조건부 UI 로직에서 삼항을 겹겹이 쓰던 코드가 경고로 바뀔 수 있음.     |
| Linter - JS 문자열 | `noTemplateCurlyInString` 추가. 템플릿 문자열 `${}` 구문을 일반 문자열에 잘못 적은 경우 잡아줌. | `"Hello ${name}"` 같은 실수들을 초기에 막아줌.                       |
| Linter - JS 문자열 | `noOctalEscape` 추가. 옛 스타일 8진수 이스케이프 사용을 막음.                    | 레거시 코드에서 오래된 이스케이프 사용 시 경고 발생.                 |
| Linter - React     | `useExhaustiveDependencies` 옵션 확장. React hook 의존성 배열 검사 행동을 더 세밀하게 조정 가능. | ESLint `react-hooks/exhaustive-deps`를 쓰던 팀은 더 정밀하게 튜닝 가능. |

이 버전도 GritQL 쿼리 스코프 확장, CLI 서머리 개선 같은 변경이 있지만, "코드 수정이 필요한 수준"은 주로 위 린트 규칙들입니다.

#### 1.9.4 (2024-10-17)

1.9.4는 거의 순수 버그픽스 버전이라, 새 규칙이나 deprecate, 의미 있는 옵션 변화는 없습니다. 실제 코드 작성 관점에서는 "업데이트하면 더 정확해진다" 정도로 이해하면 충분합니다.

### 2.2 Biome v2 - 가장 중요한 전환점

이제 진짜 중요한 지점입니다. v2부터는 "다른 린터로 바뀌었다"에 가까운 수준의 변화가 있어서, 기존 설정을 그대로 끌고 오면 결과가 많이 달라질 수 있어요.

#### 2.0.0 (2025-06-17) - Biome v2 출범

핵심 키워드: **타입 인식 린터**, **플러그인**, **새로운 글롭 엔진**, **import 정리 대공사**.

대표적인 코드 영향 요소만 뽑으면 아래처럼 정리할 수 있습니다.

| 범주                       | 변경 내용                                                                             | 코드 영향 / 체크 포인트                                                                   |
| -------------------------- | ------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| Linter - 타입 인식         | 타입 정보를 활용하는 린트 규칙 도입. TypeScript 컴파일러 없이도 타입 기반 진단 제공.   | 타입이 잘못 연결된 코드, 잘못된 제너릭 사용 등이 경고로 뜰 수 있음.                       |
| 플러그인 시스템            | Biome 플러그인으로 사용자 정의 규칙 추가 가능. GritQL과도 연계 예정.                  | ESLint 커스텀 룰을 쓰던 팀이라면 자체 Biome 플러그인을 만들어야 비슷한 경험 재현 가능.    |
| Import Organizer           | 1.x에서의 import 정렬 한계를 해결하기 위해 정렬 알고리즘이 크게 바뀜.                  | 기존 프로젝트에서 `biome check`만 돌려도 import 블록이 크게 바뀔 수 있음.                 |
| 설정 - glob 엔진           | `ignore`/`include`가 통합된 `includes`로 신규 설계. `**/` 자동 붙이기 동작 제거.      | 같은 config라도 검사되는 파일 집합이 달라질 수 있음. monorepo에서 특히 점검 필요.         |
| 설정 - 경로 기준           | v1은 "작업 디렉터리 기준", v2는 "config 파일 위치 기준"으로 해석.                     | 지금까지 잘 돌아가던 include 패턴이 다른 디렉터리를 가리킬 수 있음.                       |
| CLI - `--config-path` 제거 | `biome start`, `biome lsp-proxy`에서 `--config-path` 옵션 삭제.                       | VS Code, Zed 등에서 에디터 설정 파일에서 `biome.json` 위치를 지정하는 방식으로 변경 필요. |
| Linter - `all` 옵션 제거   | `"all"`로 모든 규칙을 켜는 방식 제거.                                                 | "추천 규칙 + 필요한 도메인만" 켜고, 개별 규칙은 명시적으로 on/off 하는 쪽으로 변경 필요.  |
| JS 문법 - `assert` → `with`| Stage 3였던 import/export `assert` 문법 지원 중단, ES 모듈의 `with` 구문으로 전환 유도. | `import foo from "./data.json" assert { type: "json" }` 코드는 `with`로 수정 권장.       |

추가로, Rome 관련 이름은 모두 Biome 용어로 바꿔야 합니다:

- `rome.json` → `biome.json`
- `// rome-ignore` → `// biome-ignore`

이건 사실상 필수 마이그레이션입니다.

## 3. 정리 - Svelte 5 + SvelteKit 2 + Biome를 쓰는 입장에서

### 지금 새로 세팅한다면

- **최소 Biome v2.3 계열**이 사실상 정답
- 이유: Svelte·Astro 정식 지원, Tailwind v4 대응, Vue·Astro까지 고려한 에코시스템 지원이 이 버전부터 제대로 들어갔기 때문

### Biome 1.x에서 바로 2.x로 올릴 때는

- `biome migrate --write`를 한 번 돌려서 설정을 자동 변환
- `files.includes`와 ignore 패턴이 "config 파일 기준 경로"로 바뀐 점을 꼭 검토 (monorepo라면 특히)

### 린트 규칙 관점에서는

- 1.9.3, 2.2.3에서 새로 들어온 규칙들이 코드 스타일에 꽤 영향을 주는 편이라, 도입 전에 팀 컨벤션과 맞는지 먼저 확인하고 켜는 게 좋음

### 요약

- **Svelte 5 + SvelteKit 2를 Biome 하나로 관리하려면** → v2.3.x
- **기존 1.x 프로젝트를 끌고 간다** → v2.0 마이그레이션 가이드에 나온 breaking change들만 정확히 이해하면, 나머지는 대부분 품질·경험 개선
