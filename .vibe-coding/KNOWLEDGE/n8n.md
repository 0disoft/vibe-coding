# n8n v1.68.0 - v1.70.1 실무 대응 전략 리포트 (2024.11 - 2025.12)

2024년 11월부터 2025년 12월까지 n8n의 변화는 **'AI 에이전트 기반 아키텍처로의 전환'**과 **'고부하 코드 실행 분리(Code Runner)'**에 집중되었습니다. 특히 v1.69.0의 서브 워크플로 디버깅 기능 도입은 복잡한 자동화 워크플로의 설계 기준을 근본적으로 바꾸었습니다.

본 리포트는 공식 문서 및 GitHub 릴리스 기록을 기반으로, 1.68.0에서 1.70.1까지의 확정적 변화와 2025년 주요 발전 방향을 분석하여 실무 전략을 제시합니다.

---

## 1. Executive Summary: 핵심 아키텍처 변화 (v1.68.0 - v1.70.1)

| 버전 | 핵심 마일스톤 | 아키텍처 영향 (Architectural Impact) |
| :--- | :--- | :--- |
| **v1.68.0** | **AI Agent Canvas** | **[UX/Model]** 워크플로 설계를 대화형 AI 에이전트 중심으로 전환 유도. 노드(Tools)를 에이전트에 종속시키는 패턴 강화. |
| **v1.69.0** | **Sub-Workflow Debugging** | **[DevOps/Logic]** 서브 워크플로(Call Workflow)의 실행 추적 허용. 워크플로의 모듈화 및 재사용성 향상. |
| **v1.70.0** | **New Canvas (Beta)** | **[UX/Design]** 새로운 캔버스 에디터 도입. 향후 워크플로 시각화 및 설계 경험의 표준화 기준 제시. |

---

## 2. Critical Action Items: 워크플로 설계 및 운영 전략

### 2-1. 모듈화 및 디버깅 전략 (v1.69.0)

* **서브 워크플로 리팩토링:** `Call Workflow` 노드를 사용하는 모든 복잡한 로직을 재검토하십시오. v1.69.0 이후에는 서브 워크플로 내부 실행을 직접 디버깅할 수 있으므로, 기존의 "로그 노드"나 "트릭"으로 추적하던 코드를 제거하고 **재사용 가능한 함수형 워크플로**로 모듈화하십시오.
* **로깅 및 알림 분기:** 실행 트레이스가 서브 워크플로 단위로 명확히 분리되므로, 실패 알림(Alerting) 조건을 '상위 워크플로 실패' 대신 **'실패한 서브 워크플로'**를 기준으로 설정하여 장애 원인 추적 정확도를 높이십시오.

### 2-2. AI 및 실행 환경 전환 (v1.68.0, v1.71.0+)

* **AI 에이전트 패턴 도입:** 신규 AI 기반 워크플로를 설계할 때는 기존의 순차적인 노드 연결 대신 **AI 에이전트 캔버스**를 활용하십시오. 이는 HTTP/DB 노드 등을 에이전트의 '도구(Tool)'로 정의하는 방식으로, LLM 기반 자동화의 예측 가능성을 높입니다.
* **코드 실행 분리 검토 (v1.71.0+):** 장기 실행되거나 고부하를 유발하는 **Code Node**가 있다면, `작업 러너(Job Runner)`를 통해 메인 n8n 프로세스와 분리하는 아키텍처를 준비하십시오. 이를 위해 Docker Compose 또는 Kubernetes 환경에 **별도의 Worker 프로세스**를 추가해야 합니다.

### 2-3. UI 및 팀 거버넌스 (v1.70.0, v1.74.0)

* **캔버스 통일:** v1.70.0의 새로운 캔버스(beta)가 앞으로의 표준 UX가 될 것입니다. 팀 내부 문서, 튜토리얼 스크린샷 등을 **새 캔버스 기준**으로 갱신하여 혼란을 방지하십시오.
* **코드 노드 일관성:** v1.74.0에서 코드 노드 편집 경험이 개편되었습니다. 팀원들이 일관된 코딩 환경을 사용하도록 내부 가이드를 업데이트하십시오.

## 4. Conclusion

n8n은 **AI 에이전트**와 **모듈화된 워크플로**를 중심으로 아키텍처를 재편하고 있습니다. **v1.69.0 이후의 서브 워크플로 디버깅**을 활용하여 워크플로의 품질을 높이고, 장기적으로 **코드 러너 분리 전략**을 통해 안정적인 대규모 자동화 인프라를 구축하십시오.
