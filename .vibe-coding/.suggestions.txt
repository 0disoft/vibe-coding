DS/콘텐츠 운영 스택 백로그(초안)

전제

- 템플릿을 기반으로 100개+ 사이트를 빠르게 찍어내야 하므로 “구성요소 조립”에 강한 DS 컴포넌트를 우선합니다.
- 작성자가 다양해질 수 있으므로 “꾸미기”보다 “구조”를 우선합니다(허용 스키마/스타일 일관성).
- TipTap 도입 가능성을 고려해, 에디터 UI는 “편집 엔진과 분리된 액션 콜백 기반”으로 설계합니다.

## 우선순위 테이블

| 우선순위 | 묶음 | 컴포넌트 | 범위 | 왜 지금 단계에서 필요한가 | 대표 사용처 |
| --- | --- | --- | --- | --- | --- |
| P0 | Editor | EditorToolbar | 블록 타입 드롭다운 + 7개 버튼 | 에디터의 최소 뼈대. 사이트가 늘수록 편집 UX 통일이 곧 운영비 절감 | 블로그/뉴스/리소스 작성 |
| P0 | Editor | EditorInlineControls | 선택 시: 굵게/기울임/링크 | 모바일/라이트 유저가 가장 자주 쓰는 3개 액션 | 블로그/뉴스/리소스 작성 |
| P0 | Content | ContentCard | 썸네일+제목+메타+태그+상태 | 목록/피드 UI는 모든 사이트에서 반복 | 블로그 목록, 리소스 목록 |
| P0 | Content | ContentMeta | 작성자/날짜/읽기시간/카테고리 | 템플릿 간 메타 정보 표현 통일 | 블로그/뉴스 상세 |
| P0 | Discovery | FilterBar | 검색+정렬+필터 슬롯 | 100개 사이트에서 “목록 탐색”이 공통 요구 | 리소스, 커뮤니티 |
| P0 | Media | MediaPicker | FileUpload 기반(업로드/선택) | 이미지 삽입은 에디터에서 가장 빠르게 필요해짐 | 썸네일/본문 이미지 |
| P1 | Editor | Quote/Callout | 인용/콜아웃 블록 | 공지/가이드/리소스에서 체감 큼 | 뉴스/가이드 |
| P1 | Editor | ClearFormatting | 형식 지우기 | 붙여넣기 품질 문제를 운영 레벨에서 줄임 | 전 콘텐츠 |
| P1 | Content | TagList | 태그 칩 리스트(+overflow) | 태그/카테고리가 많아질 때 깨짐 방지 | 블로그/리소스 |
| P1 | Content | StatusBadge | draft/published/archived | 혼자 운영해도 상태 관리는 필수 | CMS, 목록 |
| P1 | Content | ContentActions | 공유/링크복사/북마크 | 재사용되는 액션 UI 통일 | 상세 페이지 |
| P1 | Discovery | FacetFilter | 좌측/상단 패싯(카테고리/태그) | 리소스가 많아지면 필수 | 리소스/커뮤니티 |
| P2 | Analytics | ChartFrame | 제목/범례/툴팁/표 토글 | 차트 a11y는 프레임부터 SSOT를 잡아야 안전 | 대시보드 |
| P2 | Analytics | Sparkline | SVG 기반 미니 차트 | KPI 카드에서 자주 필요, 가볍게 시작 가능 | 대시보드/성과 |
| P2 | Analytics | Chart 프리미티브 | ChartFrame + Sparkline(SSOT=데이터) | “차트”를 라이브러리 종속 없이 시작하는 최소 단위 | 분석, 성과 지표 |
| P2 | Editor | Full RTE | TipTap 통합 | 스키마/저장/보안까지 함께 커져서 단계적으로 | CMS |

## EditorToolbar 기본 버튼 세트(7개)

- 블록 선택 드롭다운: 일반 문단, 제목 2, 제목 3
- 굵게
- 기울임
- 링크
- 글머리 목록
- 번호 목록
- 코드 블록

## 다음으로 붙이기 좋은 3개

- 이미지 삽입( `DsMediaPicker` 연동)
- 인용 또는 콜아웃
- 형식 지우기

- 글자 색, 배경 색, 정렬, 글자 크기
- 폰트 선택
- 임의 HTML 삽입
