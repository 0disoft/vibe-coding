# 배치 작업 프로세서 (Batch Task Processor)

`REQUESTS.md`에 등록된 미완료 작업들을 지정된 개수만큼 자동으로 식별하고 처리하기 위한 프롬프트입니다.

## 설정 (Configuration)

> [!IMPORTANT]
> 작업 시작 전 아래의 변수를 확인하거나 수정하십시오.

- **작업 개수 (Batch Size)**: 1~4개 (작업 강도와 제안 수에 따라 가변적)
- **대상 목록**: @[.vibe-coding/REQUESTS.md]
- **기본 페르소나**: @[.vibe-coding/PERSONA/LEAD_ENGINEER.md]

### 배치 사이즈 결정 기준 (Heuristics)

| 배치 사이즈 | 기준 | 비고 |
| :--- | :--- | :--- |
| **1개** | 제안 내용이 많음(~10개) 또는 작업 강도가 매우 높음 (대규모 리팩토링, 구조 변경) | 집중 처리 필요 |
| **2~3개** | 일반적인 수준 (태스크당 제안 내용 3~6개) | 기본값 범위 |
| **4개** | 제안 내용이 매우 적음(1~2개) 또는 작업 강도가 낮음 (단순 문서화, 오타 수정 등) | 병렬 처리 효율 극대화 |

## 지침 (Core Instructions)

1. **작업 식별 및 분석**: @[.vibe-coding/REQUESTS.md]에서 미완료 태스크들을 검토하고, **제안 내용의 수와 작업 강도**를 분석하여 위 테이블 기준에 따라 최적의 **N개**(Batch Size, 1~4개)를 선별하십시오.
2. **순차 및 완결 실행**:
    - 선택된 **순번**의 작업을 하나씩 순서대로 수행하십시오.
    - **개별 타당성 검토 (Crucial)**: 작업 제목이 아닌, 상세 섹션에 나열된 **각각의 제안 내용/액션 아이템 단위**로 기술적 타당성과 리스크를 검토하십시오.
    - **중요**: 검토 결과 타당하다고 판단된 제안 내용을 누락 없이 전부 구현(코드 수정)해야 해당 순번의 작업이 완료된 것으로 간주합니다. 타당하지 않은 항목은 사유와 함께 제외 내용을 기록하십시오.
    - 작업 결과에 따라 관련 파일을 직접 수정하고, 필요한 내용을 `REQUESTS.md` 하단 상세 섹션에 보강하십시오.
3. **상태 및 링크 업데이트**:
    - 리팩토링/기능 구현 태스크의 경우: 분석/제안 단계는 `🏗️ 진행 중`을 유지하고, **실제 코드 수정 및 검증이 완료**된 시점에만 `✅ 완료`로 변경하십시오.
    - 단순 정보 확인/보고 태스크의 경우: 보고서 작성이 완료되면 `✅ 완료`로 처리하십시오.
    - '비고'란에 간결한 헤더 링크(`[상세 보기](#순번-파일명-리팩토링)`)를 추가하거나 업데이트하십시오.

## 출력 형식

매 작업 완료 시마다 아래와 같이 요약 보고를 하십시오.

```markdown
### 📋 작업 완료 보고 ([작업번호])
- 대상: [파일경로/항목명]
- 수행 내용: [핵심 변경 사항 요약]
- 상태: ✅ 완료 (REQUESTS.md 업데이트됨)
```

## 작업 프로세스

1. `REQUESTS.md` 분석 및 대상 선정
2. 대상 파일 분석 및 페르소나 기반 작업 수행
3. 코드 수정 또는 제안서 생성
4. `REQUESTS.md` 상태 업데이트 및 최종 보고
