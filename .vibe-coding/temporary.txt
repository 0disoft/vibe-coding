일단 결론부터 말하면, 구조는 거의 교과서 수준이고 세부만 조금 다듬으면 될 정도야 😺
SvelteKit 튜토리얼의 `+error.svelte` 예제랑 패턴이 거의 똑같은 방식으로 `page.status`·`page.error`를 쓰고 있어서 큰 틀은 매우 정상이다. ([svelte.dev][1])

코드를 보면서 포인트별로 볼게.

```svelte
<script lang="ts">
  import { page } from '$app/state';
  import * as m from '$lib/paraglide/messages.js';
  import { localizeUrl } from '$lib/paraglide/runtime.js';
</script>

<svelte:head>
  <title>{page.status} | Error</title>
  <meta name="robots" content="noindex" />
</svelte:head>

<div
  class="flex flex-col items-center justify-center min-h-[60vh] gap-6 text-center"
  role="alert"
  aria-live="assertive"
>
  <h1 class="text-5xl font-bold tracking-tight">{page.status}</h1>
  <p class="text-xl text-muted-foreground">
    {#if page.status === 404}
      {m.error_not_found()}
    {:else}
      {page.error?.message ?? m.error_unexpected()}
    {/if}
  </p>
  <a
    href={localizeUrl('/').href}
    class="inline-flex items-center justify-center px-8 py-3 text-lg font-semibold transition-all rounded-md shadow-lg bg-primary text-primary-foreground hover:opacity-90 active:scale-95 focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2"
  >
    {m.error_go_home()}
  </a>
</div>
```

## 좋은 점부터 짚자

* `page`를 `$app/state`에서 가져와서 `page.status`, `page.error`를 직접 읽는 패턴 자체는 공식 예제랑 동일해서 전혀 문제 없음.
* `meta name="robots" content="noindex"`로 에러 페이지 인덱싱 막은 것도 좋고.
* 에러 텍스트를 전부 Paraglide 메시지로 처리해서 i18n이 정리돼 있음.
* 버튼도 시멘틱하게 `<a>`를 쓰고, 포커스 링까지 챙겨둔 게 꽤 꼼꼼하다.

이 상태에서 “망한 건 없고, 더 좋게 만들 수 있는 여지”만 손보면 될 느낌.

## 1. 4xx / 5xx 에러 메시지 분리 추천

지금은 404만 따로 처리하고, 나머지는 전부 `page.error?.message`를 노출하고 있어.

```svelte
{#if page.status === 404}
  {m.error_not_found()}
{:else}
  {page.error?.message ?? m.error_unexpected()}
{/if}
```

SvelteKit 기본 동작을 보면:

* `error(404, 'Not found')` 같이 **기대된 에러**를 던지면 그 `message`가 사용자에게 노출되고
* 코드 버그 같은 **예상 못 한 에러**는 `"Internal Error"` 같은 안전한 메시지로 교체된다 ([svelte.dev][2])

그래서 치명적인 보안 문제는 잘 안 터지지만, 그래도 관례상 이런 식으로 나누는 걸 추천해.

* 4xx: 어떤 행동을 하면 해결되는지 알려주는 쪽으로 구체적인 메시지 사용
* 5xx: 내부 사정은 숨기고, “잠깐 문제가 생겼다. 나중에 다시 시도해달라” 정도의 메시지만 노출

예를 들어 이렇게 바꿀 수 있어.

```svelte
<p class="text-xl text-muted-foreground">
  {#if page.status === 404}
    {m.error_not_found()}
  {:else if page.status >= 500}
    {m.error_unexpected()}
  {:else}
    {page.error?.message ?? m.error_unexpected()}
  {/if}
</p>
```

이러면

* `error(403, '권한이 없습니다')` 같은 건 그대로 보여줄 수 있고
* 500 같은 서버 에러는 항상 로컬라이즈된 안전한 문구로만 나가서 장기적으로도 안전해져.

## 2. `role="alert"`와 `aria-live` 강도 조정

지금은 이렇게 돼 있지.

```svelte
<div
  class="flex flex-col items-center justify-center min-h-[60vh] gap-6 text-center"
  role="alert"
  aria-live="assertive"
>
```

이 조합은 “지금 화면 전체가 급박한 경고 상황이다”라는 신호라서, SPA 네비게이션이 많은 앱에서는 화면 전환마다 스크린 리더가 굉장히 강하게 끼어드는 느낌이 될 수 있어.

에러 페이지는 어차피 “새 페이지로 완전히 들어온 상태”라서, 보조 기술 입장에서는 이미 맥락 전환이 한 번 된 셈이라 `aria-live`까지 꼭 필요하진 않다.

선택지 시뮬레이션을 해보면:

* 선택 1: `role="alert"`만 두고 `aria-live` 제거
* 선택 2: `role="alert"` 제거하고, `aria-live="polite"`로 완화
* 선택 3: 둘 다 제거하고 그냥 `<main>` 안에 `<h1>`만 명확히 두기

나는 이 페이지 구조라면 “선택 1” 정도가 적당해 보인다.

```svelte
<div
  class="flex flex-col items-center justify-center min-h-[60vh] gap-6 text-center"
  role="alert"
>
```

추가로 조금 더 친절하게 가려면 `h1`에 id를 달고 `aria-labelledby`를 써줄 수도 있어.

```svelte
<div
  class="flex flex-col items-center justify-center min-h-[60vh] gap-6 text-center"
  role="alert"
  aria-labelledby="error-title"
>
  <h1 id="error-title" class="text-5xl font-bold tracking-tight">
    {page.status}
  </h1>
  ...
</div>
```

## 3. `<title>`도 i18n에 붙여두기

현재는 고정 영문 타이틀이라서 한국어 환경에서 약간 동떨어진 느낌이 있어.

```svelte
<svelte:head>
  <title>{page.status} | Error</title>
  <meta name="robots" content="noindex" />
</svelte:head>
```

여기도 Paraglide 메시지로 가져가는 쪽이 통일감이 좋아.

예시:

```svelte
<svelte:head>
  <title>
    {#if page.status === 404}
      {m.error_title_404({ status: page.status })}
    {:else}
      {m.error_title_generic({ status: page.status })}
    {/if}
  </title>
  <meta name="robots" content="noindex" />
</svelte:head>
```

Paraglide 메시지 정의는 예를 들면:

* `error_title_404`: `"404 · 페이지를 찾을 수 없어요"`
* `error_title_generic`: `"{status} · 오류가 발생했습니다"`

이 느낌으로 맞춰두면, 나중에 언어 늘려도 타이틀이 알아서 따라와서 관리가 편해진다.

## 4. 레이아웃 관점에서 한 줄만 더 신경 쓰면 좋은 부분

`min-h-[60vh]`는 “메인 레이아웃 안에서” 에러 블록이 적당히 가운데에 오게 하려는 의도인 것 같아서 이해는 되는데, 자주 쓰이는 패턴은 두 가지야.

* 루트 `+layout.svelte`에서 `main`에 `class="flex-1"` 같은 걸 주고, 에러 페이지는 `h-full`·`min-h-full`에 맡기기
* 지금처럼 내부 컴포넌트에서 `min-h-[60vh]` 정도로 자체 높이를 확보하기

지금도 깨지는 건 아닌데, 나중에 푸터가 있는 레이아웃 구조를 쌓을 거라면 `main` 쪽에서 `flex` 레이아웃을 이미 쓰고, 에러 블럭에는 그냥 `h-full`만 주는 식으로 한 번 더 정리해봐도 좋을 것 같아.

이건 필수는 아니고 “레이아웃 레벨에서 해결할 수도 있다” 정도의 메모.

## 정리된 개선 버전 예시

위에서 말한 것들을 반영하면 대략 이런 형태가 나올 거야.

```svelte
<script lang="ts">
  import { page } from '$app/state';
  import * as m from '$lib/paraglide/messages.js';
  import { localizeUrl } from '$lib/paraglide/runtime.js';
</script>

<svelte:head>
  <title>
    {#if page.status === 404}
      {m.error_title_404({ status: page.status })}
    {:else}
      {m.error_title_generic({ status: page.status })}
    {/if}
  </title>
  <meta name="robots" content="noindex" />
</svelte:head>

<div
  class="flex flex-col items-center justify-center min-h-[60vh] gap-6 text-center"
  role="alert"
  aria-labelledby="error-title"
>
  <h1 id="error-title" class="text-5xl font-bold tracking-tight">
    {page.status}
  </h1>

  <p class="text-xl text-muted-foreground">
    {#if page.status === 404}
      {m.error_not_found()}
    {:else if page.status >= 500}
      {m.error_unexpected()}
    {:else}
      {page.error?.message ?? m.error_unexpected()}
    {/if}
  </p>

  <a
    href={localizeUrl('/').href}
    class="inline-flex items-center justify-center px-8 py-3 text-lg font-semibold transition-all rounded-md shadow-lg bg-primary text-primary-foreground hover:opacity-90 active:scale-95 focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2"
  >
    {m.error_go_home()}
  </a>
</div>
```

당장 꼭 고쳐야 하는 치명적인 버그는 없고,

* 4xx/5xx 메시지 정책
* 접근성 속성 강도
* 타이틀 i18n

이 세 가지만 정리해두면 꽤 오래 써먹을 수 있는 에러 페이지가 될 것 같아.
